# Reflection — building FuelEU Maritime with AI agents

Using AI coding agents on this assignment was less about “automatic code generation” and more about **tight feedback loops**: describe intent in layers (domain rules, ports, adapters), get a first draft, then **verify** with the compiler, linters, and tests. The largest efficiency gain was **scaffolding**: hexagonal folders, Prisma schema stubs, Express wiring, and a Vite/React shell arrived quickly enough that real effort shifted to **correctness** — formulas (energy in MJ, compliance balance, greedy pooling, compare % vs baseline), HTTP contracts, and seed data aligned with the UI.

Agents are strongest when the **constraints are explicit** (TypeScript strict mode, existing port interfaces, endpoint shapes). They are weakest when the **source of truth is external or missing** (e.g. a PDF spec not in the repo): the model may invent plausible labels or omit edge cases. Mitigation was simple but essential: **read the diff**, run **`npm run build`** and **`npm run lint`**, and add **automated tests** for pure domain logic and critical routes so regressions surface without manual clicking.

Trade-offs remained familiar. A stubbed or simplified subsystem (e.g. static route metadata on the frontend) saves time but documents a **deliberate scope cut**. Similarly, integration tests that mock use cases trade realism for **speed and determinism** — appropriate for verifying HTTP mapping and status codes, while Prisma-backed tests would belong in a separate layer if the project grew.

Overall, the workflow felt like **pair programming with a fast typist**: the human still owns architecture decisions, naming, and the evaluation checklist; the agent accelerates implementation and documentation drafts. The net gain was fewer hours on repetitive wiring and more on validating behavior against the assignment’s formulas and user-visible flows.
